语法分析器的作用

语法分析器从词法分析器中获得一个由词法单元组成的串，并且验证这个串可以由源语言的文法生成，同时能报告语法错误。

语法分析器可以被分为三种类型：

  1. 通用的：可以对任意文法进行语法分析，效率很低，不适用于编译器产品
  2. 自顶向下的：从语法分析树的顶部开始向底部构造语法分析树
  3. 自底向上的：从叶子结点向根结点的防线给构造

最高效的自顶向下和自底向上方法虽然只能处理有些文法的子类，但是其中比如LL和LR文法已经足够描述现代程序设计的大部分语言了。程序设计语言构造的语法可使用上下文无关文法 (CFG) 或BNF表示法来描述。

语法错误的处理

程序可能由不同层次的错误：

  1. 词法错误：拼写错误或者没有正确添加引号
  2. 语法错误：分号括号放错位置或者多余缺失等等
  3. 语义错误：运算符和运算分量之间类型不匹配
  4. 逻辑错误：程序员的错误推理，但是程序本身可能是良构的

实现错误处理需要：

  1. 清晰报告错误
  2. 尽快从错误中恢复并继续检测
  3. 减小处理的开销

错误恢复策略

恐慌模式的恢复

一旦发现错误就不断丢弃输入中的符号，直到找到同步词法单元集合中的符号（一般为分号或者大括号）。

这个模式的问题是可能会跳过大量输入，不检查被跳过层次的错误。

短语层次的恢复

将余下的输入的某个前缀更改成另一个串，比如将逗号替换成分号等等。但是要决定具体的替换策略，也有可能进入无限循环。

错误产生式

通过预测可能的常见错误，在当前语言的文法中加入错误产生式。

全局纠正

在理想状况下找到正确的语法，但是开销太大目前处于理论阶段。

左递归的消除

如果一个文法中有非终结符号$A$使得$A\Rightarrow A\alpha$，那么这个文法就是左递归的。

自顶向下的语法分析技术不能处理左递归的情况，因此需要消除左递归，但是自底向上的技术可以处理左递归。

自顶向下的语法分析

即为输入串构造语法分析树，从根节点开始，深度优先的创建各个结点。也可以看作是寻找输入串的最左推导。

FIRST和FOLLOW

$FIRST(\alpha)$被定义为可以从$\alpha$推导得到串的首符号的集合。如果$\alpha\Rightarrow\epsilon$，那么$\epsilon$也在$FIRST(\alpha)$中。

对于$FIRST$函数而言，必须保证对于$A\to \alpha |\beta$，$FIRST(\alpha)$和$FIRST(\beta)$不相交，使得选择具有唯一性。

$FOLLOW(A)$被定义为可能在某些句型中紧跟在$A$右边的终结符的集合。
当$A\to\alpha,\alpha\to\epsilon$时，$FOLLOW(A)$可以帮助我们选择恰当的产生式。

LL(1)文法

对于LL(1)文法，我们可以构造出不需要回溯的递归下降语法分析器。LL(1)文法的含义是从左到右扫描输入和产生最左推导，同时每一步只需要向前看一个输入符号就可以决定语法分析动作。

首先明确，左递归的文法和有二义性的文法都不是LL(1)的。

一个文法是LL(1)当且仅当对于该文法中任意两个不同的产生式$A\to\alpha |\beta$满足：

  1. $FIRST(\alpha)$和$FIRST(\beta)$是不相交的集合。
  2. 如果$\epsilon$在$FIRST(\beta)$中，那么$FIRST(\alpha)$和$FOLLOW(A)$是不相交的集合，反之同理。

对于LL(1)文法，我们只需要构造预测分析表，能保证预测分析表中每个条目都唯一的指定了一个产生式或者表明一个语法错误，就能成功解析。

自底向上的语法分析

归约

自底向上的语法分析过程可以看作是从串归约到文法开始符号S的过程。

每一次归约，就是把一个和某个产生式体相匹配的字串替换成该产生式头部的非终结符号。也就是说，一次归约就是一次推导的反向操作。

因此，自底向上的语法分析目标是反向构造一个推导过程，这个推导过程实际上对应了最右推导。

句柄的正式定义：如果$S\Rightarrow{rm}\alpha A\omega\Rightarrow{rm}\alpha\beta\omega$，那么紧跟在$\alpha$之后的$\beta$是$A\to\beta$的一个句柄。

  • 在一个最右句型中，句柄右边只有终结符号
  • 如果文法是无二义性的，那么每个句型有且只有一个句柄

移入-归约语法分析技术

我们使用一个栈来保存归约/扫描移入的文法符号。我们将用一个表格表示完整的过程，在这之前，我们需要明确：

  1. 开始时刻，栈中只包含$，而输入为$\omega$$；结束时刻，栈中只包含$S$$，而输入为$
  2. 分析动作有以下四种：
    1. 移入：将下一个输入符号移入到栈顶
    2. 归约：将句柄归约为相应的非终结符号
    3. 接受：宣布分析过程成功完成
    4. 报错：发现语法错误，调用错误恢复子程序

我们发现，句柄总是出现在栈的顶端，这使得我们可以放心的归约，而不需要到栈的中间去寻找，这只需要简单的分类讨论就可以证明。

移入-归约语法分析中的冲突

当然，并非所有的文法都可以使用移入-归约语法分析，有些文法会出现以下的问题：

  • 知道栈的所有内容也不知道该移入还是该归约
  • 有多个可能的归约方法不知道使用哪个

从技术上讲，这被称为非LR文法，这将在下面介绍。

LR语法分析技术

目前流行的自底向上的语法分析器都基于LR(k)语法分析，其中L表示最左扫描，R表示反向构造出最右推导，k表示最多向前看k个符号。

只有当k=0或k=1的时候有实践意义，当省略(k)的时候，我们假设k=1。

为什么使用LR语法分析器

  • 由表格驱动，可以自动化生成
  • 对于几乎所有的程序设计语言，只要写出上下文无关文法，就能够构造出识别该语言的LR语法分析器
  • 最通用的无回溯移入-归约分析技术
  • 能分析的文法比LL(k)文法更多

【编译原理三】词法分析

发表于 2025-02-27

本章主要讨论如何生成一个词法分析器。

词法分析器的作用

词法分析使编译的第一个阶段，其读入源程序字符串，组成词素，生成并输出一个词法单元序列。这些词法单元序列会被语法分析器作为下一步的输入。同时，词法分析器还会将词素添加到符号表中。

一般词法分析和语法分析是一趟，语法分析通过调用词法分析来不停获得词素。

词法分析器还能负责除了识别词素以外的任务，比如：

  1. 过滤源程序的注释和空白
  2. 将编译器生成的错误信息和源程序的位置联系起来
  3. 处理宏的扩展

词法单元、模式、词素

下面区别以下这些术语：

  1. 词法单元：由一个词法单元名和一个可选的属性值组成
  2. 模式：描述一个词法单元的词素可能具有的形式
  3. 词素：源程序的一个字符序列，和某个词法单元的模式匹配

总的来说，词法单元是一个抽象的类型，模式是对其的描述，词素是代码中具体的匹配字符串。

词法错误

在遇到词法错误的时候，词法分析器很难发现并改正错误，所以一般交给语法分析器去完成。

但是如果遇到所有词法单元的模式都无法匹配剩余输入，那么词法分析器就无法继续分析，我们通常采取以下方法：

  1. 删除一个字符
  2. 试图插入一个遗漏的字符
  3. 替代一个字符
  4. 交换相邻的字符
    直到能继续正确解析。

输入缓冲

现在讨论如何加快源程序的读入速度。

首先，最简单的方法：一个字符一个字符的读入是不合理的，因为这样很难区分-和->等情况。我们这里介绍缓冲区对和哨兵标记。

缓冲区对

这个机制利用两个交替读入的缓冲区，每个缓冲区容量都是N个字符，通常N是一个磁盘块的大小，如果文件剩余不足N，那么会用一个特殊字符(eof)来标记文件结束。

程序维护两个指针：

  1. lexemeBegin：指向当前词素的开始处
  2. forward：一直向前扫描直到发现某个模式被匹配，这个决定策略将在下面被讨论

一旦确定了下一个词素，词法分析器将根据这两个指针将这个词素以及其属性值记录下来。然后使lexemeBegin指针指向下一个开始的字符。

将forward指针前移的时候要检查是否到达缓冲区的末尾，如果到达末尾，就要将N个新字符读到另一个缓冲区中，并修改forward指针的指向。

哨兵标记

我们每次向前移动forward指针都需要做两次测试：

  1. 检查是否到达缓冲区的末尾
  2. 确定读入的字符是什么

【编译原理一】引论

发表于 2025-02-08 更新于 2025-02-27

编译器是一个软件翻译系统，其负责将程序设计语言翻译成计算机能执行的语言形式。

语言处理器

一个编译器可以被看作是一个程序，其作用为将某语言编写的程序翻译成等价的、另一种目标语言编写的程序，并在翻译过程中报告其在源程序中发现的错误。

解释器是另一种语言处理器，其目的和编译器一致，但是工作方式不同。两者相比：

  1. 编译器将整个源代码一次性翻译成目标机器码（通常是中间代码或直接是机器码），然后将生成的代码存储在一个独立的文件中，程序的运行时不需要源代码。编译器运行速度更快。
  2. 解释器逐行读取源代码，并将其即时翻译成机器指令进行执行，不会生成独立的目标代码。解释器错误诊断效果更好。

我们以Java语言处理器为例，其结合了编译和解释的过程。

  1. 一个Java源程序被编译成一种被称为字节码(bytecode)的形式，好处是方便字节码在不同机器之间迁移使用。
  2. 字节码由一个虚拟机进行解释执行。
  3. 有一些即时编译器还会将字节码再翻译成机器语言，然后执行程序。

一个完整的语言处理系统包含以下部分：

  1. 预处理器：将源程序聚合到一起，处理一些宏。
  2. 编译器：将源程序翻译成汇编语言程序。
  3. 汇编器：将汇编语言程序翻译成可重定位的机器代码。
  4. 链接器/加载器：生成目标机器代码，将可执行目标文件放到内存中执行。

编译器的结构

编译器由两个部分组成：

  1. 分析部分(analysis)：编译器的前端
    1. 将源程序分解为多个组成要素，并会在这些要素之上增加语法结构，从而创建该源程序的一个中间表示。
    2. 检查源程序语法构成是否正确，并给出相关信息提示。
    3. 收集有关源程序的信息，并把这些信息放在符号表中。
    4. 符号表将和中间表示形式一起传送给综合部分。
  2. 综合部分(synthesis)：编译器的后端
    1. 根据中间表示和符号表来构造用户期待的目标程序。

一般一个完整的编译流程分为以下步骤(phase)：

  1. 源程序的字符流进入词法分析器，得到符号流和符号表。符号表可能被各个步骤使用。
  2. 符号流进入语法分析得到语法树
  3. 语法树经过语义分析得到语法树
  4. 语法树经过中间代码生成器生成中间表示形式
  5. 中间表示形式经过机器无关代码优化器生成中间表示形式
  6. 中间表示形式经过代码生成器生成目标机器语言
  7. 目标机器语言经过机器相关代码优化器得到目标机器语言

其中的优化步骤一般是可选的。

词法分析(lexical analysis)

词法分析器读取源程序字符流，将其组织称为词素(lexeme)序列，对于每个词素，词法分析器产生以下词法单元(token)作为输出

其中第一个分量token-name是语法分析步骤中要使用的抽象符号，而第二个分量attribute-value是符号表中关于这个词法单元的条目。下面会举一个例子。

例如：position = initial + rate * 60 这个语句，有以下词素：

  6. 被映射为$<>$

语法分析(syntax analysis)

语法分析器使用由词法分析器生成的各个词法单元的第一个分量来创建树形的中间表示，用来给出词法单元流的语法结构。常用语法树来表示，每个结点表示一个运算，子节点表示运算的分量。

语义分析

语义分析器使用语法树和符号表的信息，来检查源程序是否和语言定义的语义一致，同时收集类型信息，进行类型检查，保证运算符和对应的分量都匹配。当然，在一般情况下，数字的类型可以自动转换。

中间代码生成

现在我们考虑将中间表示形式变成一种易于生成且易于翻译为目标语言的形式，即三地址代码。

代码优化

改进中间代码，使其更短或者能耗更低。

代码生成

将中间代码翻译成机器代码，并选择合适的寄存器和内存的位置。

符号表管理

符号表记录了一个变量名以及其各种属性，包括存储分配、类型、作用域等等，并允许编译器快速查找到每个名字对应的记录。

将多个步骤组合成趟

在具体的实现中，并不是把以上每个步骤都分得很清楚，而是会把某些步骤合并起来。这样的组合称为趟(pass)。

  1. 词法分析与语法分析合并： 在某些简单的编译器实现中，词法分析和语法分析可能会合并成一个步骤。这通常适用于语法相对简单、上下文无关的语言。词法单元（tokens）和语法树的构建可以在一个阶段中完成，节省了时间和空间。
  2. 语法分析与语义分析合并： 语法分析和语义分析可以合并成一个步骤。这种做法通常会在构建语法树的同时进行语义检查（例如类型检查、作用域分析等）。这样可以在语法树生成的同时，立即验证类型的正确性和其他语义规则，避免多次遍历代码。
  3. 中间代码生成与代码优化合并： 在某些编译器中，生成中间代码和优化代码可以在同一个阶段内进行。中间代码的生成是根据源代码构建的初步表达，而优化则可以通过遍历中间代码来减少冗余的操作，消除死代码或执行常数折叠等。这种合并能够加快编译过程，因为中间代码生成和优化通常依赖于对相同信息的多次操作。
  4. 符号表管理与语义分析合并： 符号表管理和语义分析往往可以结合在一起，因为语义分析往往依赖于符号表的信息。例如，在进行类型检查时，编译器需要同时管理符号表中的信息并检查变量是否正确使用。
  5. 多趟优化： 有些复杂的编译器在中间代码生成后，会进行多次优化，以进一步减少代码的冗余或提升执行效率。优化步骤通常会分多趟进行，例如死代码消除、循环优化、常量折叠等，可以在不同的编译阶段中多次执行。

程序设计语言的发展历程

程序设计语言有多种分类方式：

  1. 按语言的代：
    1. 第一代语言：机器语言
    2. 第二代语言：汇编语言
    3. 第三代语言：C、C++等高级程序设计语言
    4. 第四代语言：SQL等为特定应用设计的语言
    5. 第五代语言：Prolog等基于逻辑和约束的语言
  2. 按语言执行状态：
    1. 强制式语言：指明如何完成一个计算任务
    2. 声明式语言：指明要进行哪些计算
  3. 冯诺依曼语言：以冯诺依曼计算机体系结构为计算模型的程序设计语言
  4. 面向对象语言：支持面向对象编程的语言
  5. 脚本语言：具有高层次运算符的解释型语言

坐标系统

在每次顶点着色器运行后，我们必须保证所有可见顶点都是便准话设备坐标。但是在这之前显然坐标系有多种多样的选择。

对我们来说，我们将讨论以下不同的坐标系统。

  • 局部空间(Local Space，或者称为物体空间(Object Space))
  • 世界空间(World Space)
  • 观察空间(View Space，或者称为视觉空间(Eye Space))
  • 裁剪空间(Clip Space)
  • 屏幕空间(Screen Space)

局部空间

这很好理解，就是物体所在的空间，也可以理解为用建模软件来修改物体所在的坐标系。

世界空间

就是物体相对于世界零点的位置。

观察空间

此时原点变成了摄像机，方向就是摄像机的朝向。

裁剪空间

视角的范围，OpenGL期望所有的坐标都能落在一个特定的范围内，且任何在这个范围之外的点都应该被裁剪掉(Clipped)。被裁剪掉的坐标就会被忽略，所以剩下的坐标就将变为屏幕上可见的片段。这也就是裁剪空间(Clip Space)名字的由来。

一旦所有顶点被变换到裁剪空间，最终的操作——透视除法(Perspective Division)将会执行，在这个过程中我们将位置向量的x，y，z分量分别除以向量的齐次w分量；透视除法是将4D裁剪空间坐标变换为3D标准化设备坐标的过程。这一步会在每一个顶点着色器运行的最后被自动执行。

什么是纹理

纹理是一个2D图片（甚至也有1D和3D的纹理），它可以用来添加物体的细节；你可以想象纹理是一张绘有砖块的纸，无缝折叠贴合到你的3D的房子上，这样你的房子看起来就像有砖墙外表了。因为我们可以在一张图片上插入非常多的细节，这样就可以让物体非常精细而不用指定额外的顶点。

为了能够把纹理映射(Map)到三角形上，我们需要指定三角形的每个顶点各自对应纹理的哪个部分。这样每个顶点就会关联着一个纹理坐标(Texture Coordinate)。之后在图形的其它片段上进行片段插值(Fragment Interpolation)。

纹理坐标在x和y轴上，范围为0到1之间（注意我们使用的是2D纹理图像）。使用纹理坐标获取纹理颜色叫做采样(Sampling)。

GLSL

着色器的语言叫做GLSL，包含一些针对向量和矩阵操作的有用特性。

着色器的开头总是要声明版本，接着是输入和输出变量、uniform和main函数。每个着色器的入口点都是main函数。

GLSL定义了in和out关键字专门来实现这个目的。每个着色器使用这两个关键字设定输入和输出，只要一个输出变量与下一个着色器阶段的输入匹配，它就会传递下去。
